
Не стесняться добалять классы, которые не имеют прямого отношения к предметной области, для решения архитектурных задач.
## Проблема
- Какой класс должен обеспечивать реализацию паттернов High Cohesion и Low Coupling?

## Решение
- Присвоить обязанности с высокой стпенью зацепления классу, которй не предоставлет конкретного понятия из предметной области (синтезировать искусственную сущность для обеспечения выского зацепления и слабого связывания)

- Когда нужно, чтобы какие то методы жили вместе, хотя реальных классов, представляющих домен, эти методы живут отедльно, тогда нужно сделать сурогатный класс, который будет включать в себя эти методы

## Например
- Класс Storage. Его нет в бизнес моделях, но он нужен, чтобы предоставлять логику для сохранения данных в базу
- Фасад
- Репозитории

## Преимущества
- Сфабрикованный класс будет обладать низкой степенью связанности и высоким зацеплением

## Недостатки
- Данным паттерном не следует злоупотреблять иначе все функции системы превратятся в объекты. 
  
  Например паттерн Command застваляет создатаь класс с одим методом для каждой системной функции.

  Обычно лучше включить эти методы в класс доменной сущности, если нет четкой необходимости в Low Coupling, High Cohesion
