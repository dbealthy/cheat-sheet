Контроллер превращает асинхронные обращения в синхронные или наоборот

Используется в MVC модели и фреймворках для создания REST API например: Django, FastApi ...

Контроллер не содержит бизнес логику, а лишь управляет потоком выполнения
## Проблема
"Кто" должен отвечать за обработку входных 
системных событий или пользовательского ввода?

Например, асинхронные обращения от нескольких пользователей к разным частям системы, что заставляет по всюду в коде добавлять защиту от race condition, проверки на консолидацию данных  и т.д. , и усложняет и без того сложный код домена


## Решение
Обязанности по обработке системных сообщений делегировать специальному классу. Контроллер - объект, который отвечает за обработку системных событий и не относится к интерфейсу пользователя. Контроллер определяет методы для выполнения системных операций

**Контроллер все асинхронные вызовы в предметную область превращает в синхронные**


## Рекоммендации
Для различных [[Vocabulary#^7caed7|прецендентов]] логично использовать разные контроллеры (контроллеры прецендентов) - контроллеры не должны быть перегружены. Внешний контроллер может представлять всю систему целиком, если он будет не слишПрецендентком перегружен (существует всего несколько системных событий)

## Преимущества
- Удобно накапливать информацию о системных событиях (в случае если системные операции выполняются в какой то определенной последовательности)
- Удобно реализововать авторизацию, проксирование, кеширование в контроллере
- Лучше переиспользуемость (Системные события обрабатываются Контроллером, а не элементами интерфейса пользователя)
